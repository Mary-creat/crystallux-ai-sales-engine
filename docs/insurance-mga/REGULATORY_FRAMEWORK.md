# Regulatory Framework — Insurance MGA (Layer 2 Part A)

> **Audience:** Mary (MGA principal), compliance officer, regulators inspecting how AI workflows align to FSRA / PIPEDA / CASL.

## Scope

Crystallux Layer 2 Part A operates in the **insurance vertical** under three primary Canadian frameworks:

1. **FSRA Ontario** — Financial Services Regulatory Authority (insurance regulator for Ontario; equivalent provincial bodies elsewhere)
2. **PIPEDA** — Personal Information Protection and Electronic Documents Act (federal privacy)
3. **CASL** — Canadian Anti-Spam Legislation (federal commercial-electronic-message regime)

Adjacent frameworks touched in lighter form:

- **FINTRAC** — federal AML/ATF (KYC + AML risk scoring; PEP screening currently manual, Phase 6 automates)
- **OSFI** — federal prudential supervisor (relevant only to carrier-side; we are the MGA, not the carrier)
- **Provincial equivalents** — AMF (Quebec), AIC (Alberta), etc. — same compliance patterns; documentation tagged `vertical_id='insurance'` is portable per-province via per-province templates in Phase 5b.

## How each AI workflow aligns

### KYC + identity verification

| Regulation | Requirement | How Crystallux complies |
|---|---|---|
| FSRA + FINTRAC | Verify client identity before binding | `clx-mga-insurance-kyc-orchestrator-v1` creates a Stripe Identity session; `clx-mga-insurance-stripe-identity-callback-v1` records verified identity to `kyc_verifications.identity_data` (encrypted column-level) |
| FINTRAC | AML risk assessment + PEP screening | `aml_risk_score` computed from heuristics (missing fields, non-CA residency); `pep_screening_result` marked `manual_review_pending` (Phase 6 automates). Score > 75 forces human review |
| Audit | Identity verification logged | Every KYC event writes to `regulatory_audit_log` with `event_type='kyc_*'` |

### Suitability assessment + needs analysis

| Regulation | Requirement | How Crystallux complies |
|---|---|---|
| FSRA | Recommend products in client's best interest | AI ranks via `ai_match_score` (0-100) independent of carrier commission; advisor overrides logged to `compliance_reviews.human_decision` |
| FSRA | Document needs analysis basis | `suitability_assessments.client_situation` (Q&A capture) + `needs_analysis` (AI-derived gap analysis) + `ai_reasoning` form the regulatory record |
| FSRA | Client acknowledges suitability | `suitability_assessments.client_acknowledged` + `client_acknowledgment_at` |
| Audit | Each suitability decision logged | `regulatory_audit_log` events: `suitability_started`, `suitability_question_asked`, `suitability_completed` |

### Disclosure obligations

| Regulation | Disclosure required | Crystallux template |
|---|---|---|
| CASL | Express consent for commercial electronic messages | `documents/templates/insurance/casl-consent.html` |
| PIPEDA | Privacy notice (purposes, disclosures, retention, access rights) | `pipeda-privacy.html` |
| FSRA | Advisor compensation + conflict of interest disclosure | `conflict-of-interest.html` |
| FSRA | Life insurance replacement disclosure (when applicable) | `replacement-form.html` |
| FSRA | Suitability documentation record | `needs-analysis-record.html` |

All disclosures generated by `clx-mga-insurance-disclosure-generator-v1`, e-signed via Zoho Sign through `clx-mga-insurance-esignature-orchestrator-v1`, signed PDFs returned via `clx-mga-insurance-zoho-sign-callback-v1` and stored privately in R2 at `signed-disclosures/insurance/{client_id}/{envelope_id}.pdf`.

### Application + submission

| Regulation | Requirement | How Crystallux complies |
|---|---|---|
| FSRA | Application data accurate and complete | `clx-mga-insurance-application-builder-v1` auto-completes from KYC + suitability + disclosures; flags `fields_requiring_human_input` for advisor judgment |
| FSRA | Advisor reviews before submission | `policy_applications.advisor_review_status='approved'` + `locked=true` enforced by `clx-mga-insurance-application-final-review-v1` |
| FSRA | Final compliance check before binding | `clx-mga-insurance-application-final-review-v1` triggers `clx-mga-insurance-compliance-agent-v1` with `review_type='final_compliance'`; AI defaults to `requires_human_review` unless ALL gates pass |
| Audit | Submission timestamped and immutable | `policy_applications.submitted_at` set atomically; `regulatory_audit_log` event `application_submitted` |

## Audit trail (regulator-facing record)

The `regulatory_audit_log` table is **append-only** by design:

- Soft FKs (no `REFERENCES` constraint) so log entries survive related-row deletion. Cascading delete would be a regulatory failure.
- Indexed by `vertical_id`, `client_id`, `advisor_id`, `event_type`, `policy_application_id` for fast regulator-request response.
- Every AI decision is logged with `performed_by_role='agent'`. Every human override is logged with `performed_by_role='compliance_officer'` (or other) + `performed_by_user_id` for accountability.
- Retention: 7 years (FSRA + FINTRAC minimum). Schema does not auto-delete; deletion is a manual compliance-officer action after retention period (Phase 6 adds retention-policy automation).

### Sample event types logged

```
kyc_initiated, kyc_verified, kyc_failed
suitability_started, suitability_question_asked, suitability_completed
disclosure_generated, disclosure_presented, disclosure_signed, disclosure_declined
compliance_review_completed
application_built, application_review_complete, application_submitted
```

A regulator request looks like: `SELECT * FROM regulatory_audit_log WHERE client_id = $1 AND vertical_id = 'insurance' ORDER BY occurred_at;` — returns the full client lifecycle in one chronological query.

## Human-in-the-loop authority (mandatory)

The compliance_officer role retains **veto authority on every AI decision**:

- `compliance_reviews.human_decision` overrides `compliance_reviews.ai_decision`.
- `compliance_reviews.human_reviewer_id` + `human_notes` document the override basis.
- A1.1 routes ANY decision flagged `requires_human_review` to all `auth_users WHERE user_role='compliance_officer'` via email notification (Phase 5c adds dashboard alert).
- AI defaults to `requires_human_review` whenever:
  - AML risk score > 75
  - Replacement disclosure required (life insurance only)
  - Coverage > $1M
  - Vulnerable-client indicators present
  - Final pre-submission review with ANY uncertainty

This is non-optional. The AI is fast and consistent; it is **not** the regulator-facing decision-maker.

## What's NOT compliant in this build (and how Crystallux handles it)

| Gap | Risk | Mitigation now | Phase that closes it |
|---|---|---|---|
| PEP / sanctions screening manual | False negative on politically-exposed person | A1.1 routes ALL KYC reviews to `requires_human_review` if `pep_screening_result = 'manual_review_pending'` (always, until Phase 6) | Phase 6 (sanctions-list automation) |
| Carrier API submission manual | Application data could differ between Crystallux record and carrier system | Advisor manually re-enters into carrier portal; `policy_applications.application_data` is the source of truth, advisor cross-checks | Phase 6 (carrier API integration) |
| Per-province disclosure variants | Quebec / BC / etc. have provincial differences | Insurance starter library is Ontario FSRA-aligned; per-province variants ship in Phase 5b before activating non-Ontario clients | Phase 5b |
| Compliance officer dashboard | Email notifications are not a workflow | Email arrives, officer reviews via Supabase queries OR queries `compliance_reviews WHERE status='human_review_required'` directly | Phase 5c (insurer-facing mode) |
| Background checks (advisor side) | Sub-agent onboarding requires Certn checks for FSRA licensing | Currently manual; Mary handles for the 10 known agents | Phase 5b (Layer 2 Part B agent onboarding) |
